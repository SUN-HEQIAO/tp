# Membership ID Feature Implementation

**Date**: March 9, 2026

## Overview

This document describes the implementation of the **Membership ID** attribute for the contact management system. Each contact now has a unique 4-digit membership ID that is automatically generated by the system.

---

## Feature Specifications

- **Attribute**: Membership ID
- **Type**: Unique 4-digit integer
- **Range**: 1000 to 9999
- **Generation**: Automatic (system-assigned)
- **Editability**: Read-only (users cannot add or edit)
- **Behavior**: IDs only increase in magnitude (never reused)
- **Display Position**: After tags in the UI

### 1. Created MembershipId.java
**Location**: `src/main/java/seedu/address/model/person/MembershipId.java`

**Purpose**: 
- Defines a new value object to represent membership IDs
- Validates that IDs are within the 1000-9999 range
- Provides immutability guarantees

**Key Features**:
```java
public class MembershipId {
    public static final int MIN_ID = 1000;
    public static final int MAX_ID = 9999;
    public static final int MAX_CAPACITY = MAX_ID - MIN_ID + 1;  // 9000
    public static final String MESSAGE_CONSTRAINTS =
            "Membership ID must be a 4-digit integer from " + MIN_ID + " to " + MAX_ID;
    
    public final int value;
    
    public MembershipId(int id) {
        checkArgument(isValidMembershipId(id), MESSAGE_CONSTRAINTS);
        value = id;
    }
}
```

**Why**: Following the existing pattern of other attributes (Name, Phone, Email, Address), we created a dedicated class to encapsulate the membership ID logic with proper validation. The MAX_CAPACITY constant eliminates magic numbers and provides a single source of truth for the maximum number of contacts.

```java
private void updateNextMembershipId() {
    if (getPersonList().isEmpty()) {
        nextMembershipId = MembershipId.MIN_ID;
    } else {
        int maxId = Integer.MIN_VALUE;
        for (Person person : getPersonList()) {
            maxId = Math.max(maxId, person.getMembershipId().value);
        }
        nextMembershipId = maxId + 1;
    }
}

public int getNextMembershipId() {
    return nextMembershipId++;
}

public boolean canGenerateMembershipId() {
    return nextMembershipId <= MembershipId.MAX_ID;
}
```

**Why**: The AddressBook is responsible for managing the collection of persons. It tracks the next available membership ID to ensure uniqueness and proper incrementing. When loading existing data, it finds the maximum existing ID and sets the counter to max + 1.

### 6. Updated AddCommand.java
**Location**: `src/main/java/seedu/address/logic/commands/AddCommand.java`

**Changes**:
- Modified `execute()` method to generate a new membership ID from the model
- Creates a new Person with the generated ID before adding to the address book
- Added capacity check to prevent adding contacts when address book is full
- Added error message for maximum capacity reached

**Implementation**:
```java
public static final String MESSAGE_SUCCESS = "New person added: %1$s";
public static final String MESSAGE_DUPLICATE_PERSON = "This person already exists in the address book";
public static final String MESSAGE_ADDRESS_BOOK_FULL =
        "Cannot add more contacts. Address book has reached its maximum capacity of "
        + MembershipId.MAX_CAPACITY + " contacts.";

@Override
public CommandResult execute(Model model) throws CommandException {
    requireNonNull(model);

    if (model.hasPerson(toAdd)) {
        throw new CommandException(MESSAGE_DUPLICATE_PERSON);
    }

    // Check if address book has reached maximum capacity
    if (!model.canGenerateMembershipId()) {
        throw new CommandException(MESSAGE_ADDRESS_BOOK_FULL);
    }

    // Generate new membership ID and create person with it
    int newMembershipId = model.getNextMembershipId();
    Person personWithId = new Person(toAdd.getName(), toAdd.getPhone(), toAdd.getEmail(),
            toAdd.getAddress(), toAdd.getTags(), new MembershipId(newMembershipId));

    model.addPerson(personWithId);
    return new CommandResult(String.format(MESSAGE_SUCCESS, Messages.format(personWithId)));
}
```

**Why**: This is where the automatic ID generation happens. When a user adds a contact, AddCommand gets the next available ID from the model and creates a Person with that ID before adding it to the address book. The capacity check now uses `model.canGenerateMembershipId()` so command logic is based on ID availability (source of truth), not UI filter state.

## Technical Details

### ID Generation Algorithm
1. Start with `nextMembershipId = MembershipId.MIN_ID` (1000)
2. When adding a person:
   - Check if address book is at capacity (9000 contacts)
   - If not, get current `nextMembershipId`
   - Assign to new person
   - Increment counter
3. When loading data:
   - Find maximum existing ID
   - Set `nextMembershipId = max + 1`

### Capacity Limit Handling
- **Maximum contacts**: 9000 (defined by `MembershipId.MAX_CAPACITY`)
- **When limit is reached**: User receives friendly error message
- **Error message**: "Cannot add more contacts. Address book has reached its maximum capacity of 9000 contacts."
- **No magic numbers**: All capacity-related values use `MembershipId.MAX_CAPACITY` constant

### Validation Rules
- Must be an integer
- Must be between `MembershipId.MIN_ID` (1000) and `MembershipId.MAX_ID` (9999) inclusive
- Must be unique (enforced by sequential generation)

### Data Persistence
- Stored in JSON as `"membershipId": 1000`
- Validated on load to ensure within valid range
- Application fails gracefully if invalid IDs are detected

## Benefits

1. **Uniqueness**: Each contact has a guaranteed unique identifier
2. **Simplicity**: No user input required - fully automatic
3. **Consistency**: IDs follow a predictable sequential pattern
4. **Traceability**: IDs never change, making contacts easy to reference
5. **Real-world Pattern**: Mimics how membership systems work in practice

## Limitations

- Maximum 9000 unique contacts (IDs from 1000 to 9999)
- IDs are never reused, even if a contact is deleted
- No customization of ID format (always 4 digits)
- No backward compatibility with data files without membership IDs

## Code Patterns Used

### Value Object Pattern
- MembershipId is a value object (immutable, validated)

### Auto-increment Pattern
- Sequential ID generation with atomic increment
- Similar to database auto-increment primary keys

- Model interface exposes `getNextMembershipId()` and `canGenerateMembershipId()`
- Hides AddressBook implementation details from commands

- Person objects are immutable

## Troubleshooting

### Issue: "Person's membershipId field is missing!"
**Cause**: Old JSON data without membership IDs  
**Solution**: Check that AddressBook initialization sets `nextMembershipId = MembershipId.MIN_ID`

### Issue: IDs not incrementing properly
**Cause**: nextMembershipId not updated after loading data  
**Solution**: Verify `updateNextMembershipId()` is called in `resetData()`

### Issue: Duplicate membership IDs
**Cause**: Counter not properly initialized  
**Solution**: Check that AddressBook initialization sets `nextMembershipId = 1000`
